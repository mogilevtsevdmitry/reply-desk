# prompts/ — контракт сборки system-промпта

Эта папка — источник промптов для AI-слоя (см. docs/02-DEVELOPER.md, раздел
«Сборка промпта»). Файлы — статичные md-шаблоны; всё динамическое подставляет код
воркера. Промпты написаны на русском, императивно, без дублирования JSON-формата —
структуру вывода обеспечивает tool use (`emit_result`, схема из zod), а требования
к содержанию полей описаны в base.md.

## Порядок конкатенации system-промпта

Строго в этом порядке, разделитель — пустая строка:

1. `base.md` — целиком.
2. `niches/{niche}.md` — файл по `Company.niche` (enum в нижнем регистре:
   `SALON → salon.md`, … `OTHER → other.md`). Целиком.
3. `platforms.md` — **только секция площадки отзыва**. Код вырезает секцию по
   заголовку-маркеру `## {ReviewSource}` (маркеры совпадают со значениями enum:
   `## YANDEX_MAPS`, `## TWOGIS`, `## OZON`, `## WILDBERRIES`, `## OTHER`) —
   от маркера до следующего `## ` или конца файла, плюс общая преамбула файла
   (текст до первого `## `).
4. `fake-detection.md` — целиком.
5. **Блок тона компании** — генерируется кодом из `Company`:

   ```
   # Компания

   Название: {name}
   Тон бренда по умолчанию: {soft|neutral|confident} ({подпись из COPY.md})
   Чего избегать в ответах (требование владельца): {avoid или «не задано»}

   Примеры текстов компании (переними лексику и манеру):
   1. {examples[0]}
   ...
   ```

   Поля `avoid` и `examples` опциональны: пустые — строка/секция опускается,
   заголовок блока остаётся всегда.
6. **Блок похожих отзывов** — только если pg_trgm вернул кандидатов (шаг 2 пайплайна):

   ```
   # Кандидаты похожих отзывов этой компании

   Сравни по смыслу жалобы (правила — в блоке «Классификация»).
   - id: {review.id} | дата: {createdAt YYYY-MM-DD} | категория: {category или «—»}
     текст: {rawText, усечённый до ~500 символов}
   ```

   Если кандидатов нет — блок не добавляется вовсе (base.md трактует отсутствие
   блока как isRepeat = false).

## User-сообщение

Тоже собирает код:

```
Площадка: {ReviewSource}
Оценка клиента: {rating} из 5   ← строка опускается, если rating не передан

<review>
{rawText как есть, без обработки}
</review>
```

Текст отзыва — только внутри `<review>...</review>` и нигде больше (anti-injection
контракт из base.md и docs/04-SECURITY.md). Экранировать содержимое не нужно,
но код должен отклонять/нейтрализовать строку `</review>` внутри rawText
(например, заменой на `<\/review>`), чтобы пользователь не мог закрыть разделитель.

## Что обеспечивает код (не промпты)

- JSON-структуру результата — tool use с одной tool `emit_result` (zod → JSON Schema).
  В промптах формат не описан и описываться не должен.
- Валидацию `similarReviewIds`: только подмножество id переданных кандидатов,
  лишние отбрасываются.
- Один повторный запрос при невалидном JSON с текстом ошибки валидации, затем FAILED.
- Усечение текстов кандидатов и лимит длины отзыва (4000).

## Правила сопровождения

- Заголовки-маркеры `## ` в `platforms.md` — программный контракт: не переименовывать
  и не вкладывать `## ` внутрь секций без правки кода-резчика.
- Терминология текстов — по docs/content/COPY.md (тона «мягкий/нейтральный/уверенный»,
  словесные метки severity, формулировки флага заказного отзыва).
- Имена файлов ниш — программный контракт с enum `Niche`.
- Изменения смысловых правил (критерии severity, шкала компенсаций, правила флага) —
  фиксировать в docs/DECISIONS.md.
