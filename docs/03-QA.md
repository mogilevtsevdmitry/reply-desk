# 03 — ТЗ: Тестировщик

Прочитай 00-OVERVIEW.md и 02-DEVELOPER.md (контракты — источник истины).
Работаешь с готовым кодом. LLM в тестах ВСЕГДА мокается (`FakeLlmProvider`
с детерминированными ответами + режимы: невалидный JSON, таймаут, ошибка сети).

## Стек тестов

- Юнит и интеграционные (API): Jest + Supertest, Testcontainers (PostgreSQL+pgvector, Redis)
- E2E (UI): Playwright против docker compose окружения
- Команды: `pnpm test`, `pnpm test:e2e` — обе должны работать из чистого клона

## Матрица обязательных проверок

### Auth
- регистрация: успех / дубль email (409) / слабый пароль (422)
- логин: успех / неверный пароль (401, одинаковое сообщение для «нет юзера» и «не тот пароль»)
- refresh: ротация токена, повторное использование старого refresh → 401 + ревокация
  всех refresh-токенов пользователя
- доступ к /reviews без токена → 401, с протухшим access → 401

### Company / онбординг
- создание компании один раз; повторный POST /company → 409
- toneOfVoice: больше 3 примеров или пример > 1000 символов → 422

### Reviews + лимиты
- POST /reviews: пустой текст / > 4000 символов / неверный source → 422
- happy path: 202, Generation в статусе PENDING, job в очереди
- лимит FREE=10: 11-я генерация в периоде → 402 LIMIT_EXCEEDED
- лимит резервируется при POST /reviews; FAILED-генерация возвращает лимит
  (декремент-компенсация — проверить UsageCounter до/после)
- retry: POST /reviews/:id/retry для FAILED → 202, новая генерация, лимит резервируется
  заново; retry для генерации не в FAILED → 409
- конкурентность: 5 параллельных POST при остатке лимита 1 → ровно одна проходит
  (атомарность инкремента)
- изоляция тенантов: пользователь A не читает review пользователя B (404, не 403)

### Пайплайн генерации (FakeLlmProvider)
- статусы идут по порядку PENDING→ANALYZING→GENERATING→DONE и приходят в SSE
- payload сохраняется во все 4 поля, Review.category/severity обновлены
- невалидный JSON от LLM → один ретрай → при повторе FAILED + error
- таймаут LLM → FAILED, SSE отдаёт финальное событие, фронт не виснет
- похожие отзывы: посеять 3 текстово похожих отзыва (pg_trgm) → similarReviewIds не пуст,
  isRepeat=true; similarReviewIds содержит только id отзывов своего тенанта

### E2E (Playwright)
1. Полный happy path: регистрация → онбординг 3 шага → вставка отзыва → анимация →
   4 карточки → копирование ответа (проверить clipboard) → отзыв появился в истории
2. Фильтры истории: по источнику и severity возвращают корректное подмножество
3. Ошибка генерации: мок FAILED → пользователь видит понятное сообщение и кнопку
   «Повторить»; клик запускает новую генерацию (POST /reviews/:id/retry) до DONE
4. 402 лимит → экран апгрейда
5. Прогон happy path на viewport 360px

### Нефункциональные
- время ответа POST /reviews (без LLM) < 300ms p95 на локальном стенде
- SSE-соединение переживает 60с генерации (keep-alive)
- XSS-проба: отзыв с `<script>`, markdown-инъекцией, эмодзи — рендер безопасен,
  копирование отдаёт исходный текст

## Артефакты

- Тесты в репо (apps/api/test, apps/web/e2e)
- `docs/qa/BUGS.md` — каждый баг: шаги, ожидаемое/фактическое, severity (blocker/major/minor)
- `docs/qa/REPORT.md` — покрытие матрицы (пройдено/упало/заблокировано), вердикт:
  готов ли продукт к передаче безопаснику

Definition of Done: вся матрица автоматизирована, blocker/major багов нет либо
заведены и переданы разработчику с воспроизведением.
