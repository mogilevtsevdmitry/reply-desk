# DECISIONS — журнал архитектурных решений (ADR)

Формат: контекст / решение / последствия. Дополняется всеми агентами по ходу работы.

## ADR-001: Похожие отзывы через pg_trgm вместо embeddings (2026-06-11)

**Контекст.** Исходное ТЗ требовало `embed()` у `AnthropicProvider` и pgvector, но у
Anthropic нет embeddings API. Добавлять второго AI-провайдера (Voyage/OpenAI) не хотим —
зафиксировано: Anthropic API остаётся единственным внешним AI-сервисом.

**Решение.** Поиск похожих отзывов — триграммный (`pg_trgm`, встроен в PostgreSQL):
параметризованный запрос `similarity("rawText", $text) > SIMILARITY_THRESHOLD` (env,
default 0.3), top-5 кандидатов в рамках тенанта. Кандидаты передаются в промпт,
финальное решение `isRepeat` / `similarReviewIds` принимает LLM в общем вызове генерации.
Колонка `embedding` и расширение `vector` удалены из схемы.

**Последствия.** Нет второго API-ключа и затрат на embeddings; фича работает на
seed-данных (текстовая похожесть, не векторная); образ postgres — стандартный
`postgres:16-alpine`. Минус: триграммы ловят лексическую похожесть, а не семантическую —
перефразированные жалобы могут не находиться. Если станет критично, вернём векторный
поиск отдельным ADR (схема это допускает: добавить колонку миграцией).

## ADR-002: Лимиты по модели «резервирование при POST + компенсация при FAILED» (2026-06-11)

**Контекст.** Исходное ТЗ противоречило себе: инкремент UsageCounter был и в воркере
(шаг DONE), и «проверка в transaction при POST»; QA-тест на конкурентность (5 параллельных
POST при остатке 1 → ровно одна проходит) выполним только при атомарном списании на POST.

**Решение.** Атомарный инкремент UsageCounter в той же транзакции, что создание
Review + Generation (POST /reviews и POST /reviews/:id/retry). При FAILED воркер делает
атомарный декремент-компенсацию (не ниже 0). Воркер при DONE лимит не трогает.

**Последствия.** Конкурентные запросы не пробивают лимит; FAILED не «съедает» генерацию.
Кратковременно счётчик может быть «занят» генерацией, которая упадёт — приемлемо для MVP.

## ADR-003: Ретрай упавшей генерации — POST /reviews/:id/retry (2026-06-11)

**Контекст.** E2E-сценарий требует кнопку «Повторить» при FAILED, но эндпоинта ретрая
в контрактах не было, а Generation связан с Review 1:1.

**Решение.** `POST /reviews/:id/retry`: допустим только для Generation в статусе FAILED
(иначе 409), заново резервирует лимит (402 при исчерпании), сбрасывает статус в PENDING
и кладёт job в очередь. Generation остаётся 1:1 с Review (переиспользуется та же запись).

**Последствия.** Нет дублей Review при повторе; история попыток не хранится (вне scope MVP).

## ADR-004: Аутентификация SSE — fetch-based поток с заголовком Authorization (2026-06-11)

**Контекст.** Access-токен живёт в памяти фронта; нативный `EventSource` не умеет
передавать заголовки, токен в query — утечка в логи/историю.

**Решение.** SSE-подписка реализуется через `fetch` + ReadableStream с заголовком
`Authorization: Bearer`. Сервер проверяет принадлежность generation тенанту до подписки.
Токен в query-параметрах запрещён.

**Последствия.** Чуть больше кода на фронте (парсинг event-stream вручную или через
обёртку), зато единая модель аутентификации для всех запросов.

## ADR-005: Перевыпуск access-токена после онбординга (2026-06-11)

**Контекст.** JWT содержит `sub, companyId`, но компания создаётся после регистрации —
токен, выданный на регистрации, содержит `companyId=null`, а на companyId из JWT
построена вся изоляция тенантов.

**Решение.** `POST /company` возвращает `{ company, accessToken }` — новый access-токен
с заполненным companyId. Фронт заменяет токен в памяти перед redirect в /app.

**Последствия.** Никаких «полупустых» JWT в работе с /reviews; refresh-флоу не меняется
(при refresh токен выпускается уже с актуальным companyId из БД).

## ADR-006: Шрифтовая пара Prata + Golos Text через Google Fonts CDN (2026-06-11)

**Контекст.** ТЗ дизайна требует характерный display-шрифт + нейтральный текстовый,
оба с кириллицей, «не Inter+Inter»; Google Fonts по ссылке допустимы. Self-hosting
шрифтов в ТЗ не оговорён.

**Решение.** Display — Prata (антиква, спроектирована под кириллицу, одно начертание
400), текст — Golos Text (400/500/600, кириллица нативная). Подключение —
`<link>` на Google Fonts с `display=swap`; фолбэки Georgia / Arial. Прототипы и
будущий фронт используют один и тот же `<link>`.

**Последствия.** Ноль файлов шрифтов в репо; у Prata нет жирных начертаний —
иерархия заголовков строится размером, что зафиксировано в типографической шкале.
Если для прода потребуется независимость от CDN (152-ФЗ / офлайн), шрифты
переносятся в self-hosted woff2 отдельным шагом без смены пары.

## ADR-007: Прототипы делят общий tokens.css относительной ссылкой (2026-06-11)

**Контекст.** ТЗ требует «самодостаточные HTML+CSS, открываются без сборки», но
токены вынесены в отдельный артефакт tokens.css. Полное инлайнирование токенов в
каждый файл размножило бы источник истины на семь копий.

**Решение.** Каждый прототип подключает `<link rel="stylesheet" href="tokens.css">`
(относительный путь внутри docs/design/ работает с file://). Компонентные стили
дублируются инлайном в каждом экране — это прототипы, а не продакшен-код;
единственный источник истины для значений — tokens.css.

**Последствия.** Правка палитры/таймингов в одном месте меняет все экраны; файлы
открываются двойным кликом. Дублирование компонентного CSS между экранами
осознанное и в Next.js не переносится (разработчик соберёт компоненты из ui-kit).

## ADR-008: Каркас приложения — сайдбар 232px, на мобильном — верхняя панель (2026-06-11)

**Контекст.** ТЗ описывает три раздела (генерация, история, настройки) и лимит
тарифа, но молчит о навигационном каркасе.

**Решение.** Постоянный левый сайдбар (232px): логотип, три пункта навигации,
внизу — счётчик лимита «7 из 10» с прогресс-баром (лимит видим постоянно, без
заходов в настройки). На ≤880px сайдбар складывается в горизонтальную верхнюю
панель, счётчик сжимается до числа.

**Последствия.** Простейший паттерн, переносится в Next.js одним layout-компонентом;
брейкпоинт один (880px), покрывает обязательные 360/1440. Если разделов станет
больше пяти — каркас пересматривается отдельным ADR.

## Зафиксированные мелочи

- Повторное использование старого refresh-токена → ревокация **всех** refresh-токенов
  пользователя (отдельное поле «семьи» в схеме не вводим).
- Код 402 для LIMIT_EXCEEDED оставлен как есть (зафиксирован в ТЗ и тестах).
- Риск 152-ФЗ (тексты отзывов с ПД уходят в зарубежный LLM) принят для MVP; смягчение —
  провайдер-агностичный адаптер `LlmProvider`, позволяющий заменить провайдера без
  переписывания доменного кода. Не блокер, отмечен как продуктовый риск.
