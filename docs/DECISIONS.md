# DECISIONS — журнал архитектурных решений (ADR)

Формат: контекст / решение / последствия. Дополняется всеми агентами по ходу работы.

## ADR-001: Похожие отзывы через pg_trgm вместо embeddings (2026-06-11)

**Контекст.** Исходное ТЗ требовало `embed()` у `AnthropicProvider` и pgvector, но у
Anthropic нет embeddings API. Добавлять второго AI-провайдера (Voyage/OpenAI) не хотим —
зафиксировано: Anthropic API остаётся единственным внешним AI-сервисом.

**Решение.** Поиск похожих отзывов — триграммный (`pg_trgm`, встроен в PostgreSQL):
параметризованный запрос `similarity("rawText", $text) > SIMILARITY_THRESHOLD` (env,
default 0.3), top-5 кандидатов в рамках тенанта. Кандидаты передаются в промпт,
финальное решение `isRepeat` / `similarReviewIds` принимает LLM в общем вызове генерации.
Колонка `embedding` и расширение `vector` удалены из схемы.

**Последствия.** Нет второго API-ключа и затрат на embeddings; фича работает на
seed-данных (текстовая похожесть, не векторная); образ postgres — стандартный
`postgres:16-alpine`. Минус: триграммы ловят лексическую похожесть, а не семантическую —
перефразированные жалобы могут не находиться. Если станет критично, вернём векторный
поиск отдельным ADR (схема это допускает: добавить колонку миграцией).

## ADR-002: Лимиты по модели «резервирование при POST + компенсация при FAILED» (2026-06-11)

**Контекст.** Исходное ТЗ противоречило себе: инкремент UsageCounter был и в воркере
(шаг DONE), и «проверка в transaction при POST»; QA-тест на конкурентность (5 параллельных
POST при остатке 1 → ровно одна проходит) выполним только при атомарном списании на POST.

**Решение.** Атомарный инкремент UsageCounter в той же транзакции, что создание
Review + Generation (POST /reviews и POST /reviews/:id/retry). При FAILED воркер делает
атомарный декремент-компенсацию (не ниже 0). Воркер при DONE лимит не трогает.

**Последствия.** Конкурентные запросы не пробивают лимит; FAILED не «съедает» генерацию.
Кратковременно счётчик может быть «занят» генерацией, которая упадёт — приемлемо для MVP.

## ADR-003: Ретрай упавшей генерации — POST /reviews/:id/retry (2026-06-11)

**Контекст.** E2E-сценарий требует кнопку «Повторить» при FAILED, но эндпоинта ретрая
в контрактах не было, а Generation связан с Review 1:1.

**Решение.** `POST /reviews/:id/retry`: допустим только для Generation в статусе FAILED
(иначе 409), заново резервирует лимит (402 при исчерпании), сбрасывает статус в PENDING
и кладёт job в очередь. Generation остаётся 1:1 с Review (переиспользуется та же запись).

**Последствия.** Нет дублей Review при повторе; история попыток не хранится (вне scope MVP).

## ADR-004: Аутентификация SSE — fetch-based поток с заголовком Authorization (2026-06-11)

**Контекст.** Access-токен живёт в памяти фронта; нативный `EventSource` не умеет
передавать заголовки, токен в query — утечка в логи/историю.

**Решение.** SSE-подписка реализуется через `fetch` + ReadableStream с заголовком
`Authorization: Bearer`. Сервер проверяет принадлежность generation тенанту до подписки.
Токен в query-параметрах запрещён.

**Последствия.** Чуть больше кода на фронте (парсинг event-stream вручную или через
обёртку), зато единая модель аутентификации для всех запросов.

## ADR-005: Перевыпуск access-токена после онбординга (2026-06-11)

**Контекст.** JWT содержит `sub, companyId`, но компания создаётся после регистрации —
токен, выданный на регистрации, содержит `companyId=null`, а на companyId из JWT
построена вся изоляция тенантов.

**Решение.** `POST /company` возвращает `{ company, accessToken }` — новый access-токен
с заполненным companyId. Фронт заменяет токен в памяти перед redirect в /app.

**Последствия.** Никаких «полупустых» JWT в работе с /reviews; refresh-флоу не меняется
(при refresh токен выпускается уже с актуальным companyId из БД).

## ADR-006: Шрифтовая пара Prata + Golos Text через Google Fonts CDN (2026-06-11)

**Контекст.** ТЗ дизайна требует характерный display-шрифт + нейтральный текстовый,
оба с кириллицей, «не Inter+Inter»; Google Fonts по ссылке допустимы. Self-hosting
шрифтов в ТЗ не оговорён.

**Решение.** Display — Prata (антиква, спроектирована под кириллицу, одно начертание
400), текст — Golos Text (400/500/600, кириллица нативная). Подключение —
`<link>` на Google Fonts с `display=swap`; фолбэки Georgia / Arial. Прототипы и
будущий фронт используют один и тот же `<link>`.

**Последствия.** Ноль файлов шрифтов в репо; у Prata нет жирных начертаний —
иерархия заголовков строится размером, что зафиксировано в типографической шкале.
Если для прода потребуется независимость от CDN (152-ФЗ / офлайн), шрифты
переносятся в self-hosted woff2 отдельным шагом без смены пары.

## ADR-007: Прототипы делят общий tokens.css относительной ссылкой (2026-06-11)

**Контекст.** ТЗ требует «самодостаточные HTML+CSS, открываются без сборки», но
токены вынесены в отдельный артефакт tokens.css. Полное инлайнирование токенов в
каждый файл размножило бы источник истины на семь копий.

**Решение.** Каждый прототип подключает `<link rel="stylesheet" href="tokens.css">`
(относительный путь внутри docs/design/ работает с file://). Компонентные стили
дублируются инлайном в каждом экране — это прототипы, а не продакшен-код;
единственный источник истины для значений — tokens.css.

**Последствия.** Правка палитры/таймингов в одном месте меняет все экраны; файлы
открываются двойным кликом. Дублирование компонентного CSS между экранами
осознанное и в Next.js не переносится (разработчик соберёт компоненты из ui-kit).

## ADR-008: Каркас приложения — сайдбар 232px, на мобильном — верхняя панель (2026-06-11)

**Контекст.** ТЗ описывает три раздела (генерация, история, настройки) и лимит
тарифа, но молчит о навигационном каркасе.

**Решение.** Постоянный левый сайдбар (232px): логотип, три пункта навигации,
внизу — счётчик лимита «7 из 10» с прогресс-баром (лимит видим постоянно, без
заходов в настройки). На ≤880px сайдбар складывается в горизонтальную верхнюю
панель, счётчик сжимается до числа.

**Последствия.** Простейший паттерн, переносится в Next.js одним layout-компонентом;
брейкпоинт один (880px), покрывает обязательные 360/1440. Если разделов станет
больше пяти — каркас пересматривается отдельным ADR.

## ADR-009: Копидек закрывает значения enum, пропущенные в прототипах (2026-06-11)

**Контекст.** Прототипы дизайнера покрывают не все значения enum из схемы данных:
нет ниши MEDICAL (7-е значение Niche), площадки OTHER (5-е значение ReviewSource)
и категории OTHER (6-е значение Category). ТЗ контент-рерайтера требует подписи
всех значений; без них часть доменной модели недостижима из UI.

**Решение.** COPY.md определяет недостающие подписи: «Медицинская клиника»,
«Другая площадка» (+ пометка карточки ответа «без привязки к площадке» вместо
«адаптировано под…»), «Другое» для категории. Прототипы не правятся — копидек
источник истины, разработчик добавляет элементы по нему.

**Последствия.** UI и схема данных согласованы; нишевый промпт medical.md и
универсальный путь «другая площадка» доступны пользователю. Визуальный стиль
новых чипов/опций берётся из существующих (для source-other — нейтральная точка
без брендового цвета).

## ADR-010: Примеры текстов — три отдельных поля по 1000 символов (2026-06-11)

**Контекст.** Контракт хранит примеры тона как `examples: string[]` (до 3, по
1000 символов), а прототип шага 3 онбординга и настроек показывает одну textarea.
Разбор одной textarea на массив (по пустым строкам?) — неочевидное поведение
и невалидируемая по-штучно длина.

**Решение.** В копидеке — три отдельных необязательных поля «Пример 1/2/3»
со счётчиком «{n} / 1000» и ошибкой превышения; плейсхолдер только у первого.
То же на экране настроек.

**Последствия.** Валидация тривиальна и совпадает с контрактом 1:1; UX-цена —
чуть больше вертикального места. Прототип расходится с продуктом — зафиксировано
в списке расхождений COPY.md.

## ADR-011: Поле «Чего избегать в ответах» остаётся и живёт в toneOfVoice (2026-06-11)

**Контекст.** Прототипы (онбординг шаг 2, настройки) содержат поле «Чего избегать
в ответах», но комментарий к схеме описывает toneOfVoice как `{ tone, examples }` —
поля нет ни в ТЗ контента, ни в контракте явно.

**Решение.** Поле сохраняется (необязательное): это самый дешёвый способ дать
владельцу контроль над запретами («не обещать возврат денег»). Хранится строкой
`avoid` внутри JSON-колонки `Company.toneOfVoice` — миграция не нужна; строка
добавляется в блок тона system-промпта.

**Последствия.** Прототип, копидек и промпт согласованы; контракт zod дополняется
опциональным полем. Если поле не будет реализовано разработчиком, ключи
`onb-avoid-*` / `settings-avoid-label` просто не используются — остальной копидек
не затронут.

## ADR-012: Содержимое 402-экрана апгрейда (заглушка без оплаты) (2026-06-11)

**Контекст.** ТЗ требует 402-экран с описанием тарифов START/BUSINESS и
CTA-заглушкой, но прототипа экрана нет, цены тарифов нигде не зафиксированы,
оплата вне scope MVP.

**Решение.** Экран описывает тарифы только через известные факты — лимиты
генераций из env (START = 100, BUSINESS = 1000), без цен и выдуманных фич.
CTA-заглушка — «Сообщить мне, когда тарифы откроются»: по нажатию только тост
«Записали — напишем на {email}…», без бэкенд-логики (почта уже есть в аккаунте).
Вторичное действие — «Вернуться к истории».

**Последствия.** Ни одного выдуманного обещания; когда появится биллинг, экран
дополняется ценами и реальным CTA без смены структуры. Тон экрана — спокойный:
показываем дату обнуления счётчика, чтобы снять ощущение тупика.

## ADR-013: Роль исполнителя внутренней задачи — свободный текст с рекомендованным набором (2026-06-11)

**Контекст.** ТЗ контента требует «подписи ролей» для поля «Кому» внутренней
задачи, но enum ролей в схеме нет — `assigneeRole` приходит свободным текстом
из LLM, а командные роли вне scope MVP.

**Решение.** Роли остаются свободным текстом. Копидек фиксирует рекомендованный
набор для промптов (Владелец, Управляющий, Администратор, Старший мастер или
специалист), нишевые промпты конкретизируют его («старший колорист», «старший
врач»). UI выводит значение как есть.

**Последствия.** Никакого enum и миграций; согласованность ролей обеспечивается
промптами, а не валидацией. Если в пост-MVP появятся командные роли, набор
превратится в enum отдельным ADR.

## ADR-014: Критерии severity и шкала компенсаций зафиксированы в base.md (2026-06-11)

**Контекст.** ТЗ задаёт шкалу severity 1–5 и словесные метки (COPY.md: фоновая…критичная),
но не определяет критерии уровней и связь компенсации с серьёзностью. Без явных критериев
LLM выставляет severity нестабильно, а рекомендации по компенсации дрейфуют от «ничего»
до «вернуть деньги» на одинаковых отзывах.

**Решение.** В prompts/base.md зафиксированы: (а) критерии каждого уровня severity,
согласованные со словесными метками COPY.md (5 = вред здоровью/имуществу, угроза
надзором/судом; 4 = уход клиента, требование возврата, грубость, повторный сбой; и т.д.);
(б) шкала компенсаций, привязанная к severity: 1–2 — ничего/символическая, 3 — скидка
или бесплатное исправление, 4 — исправление + скидка (деньги — решение владельца),
5 — без заочной компенсации, только личный контакт владельца. Нишевые файлы поднимают
порог для отраслевых рисков (санитария, боль/осложнения, безопасность авто — минимум 4).

**Последствия.** Консистентность severity между генерациями и нишами; LLM никогда не
обещает возврат денег от имени компании — денежные решения всегда эскалируются владельцу.
Изменение шкалы — правка одного файла + новый ADR.

## ADR-015: Программный контракт промптов — маркеры секций и нейтрализация разделителя (2026-06-11)

**Контекст.** ТЗ говорит «platforms.md (секция площадки)», но не задаёт формат вырезания
секции; anti-injection требование (04-SECURITY.md) задаёт разделители `<review>…</review>`,
но не закрывает случай, когда злоумышленник вставляет `</review>` в сам текст отзыва.

**Решение.** (а) Секции platforms.md размечены заголовками `## {ReviewSource}` — маркеры
буквально равны значениям enum (`## YANDEX_MAPS` и т.д.); код вырезает от маркера до
следующего `## ` или конца файла и добавляет преамбулу файла (текст до первого `## `).
(б) Код сборки user-сообщения обязан нейтрализовать строку `</review>` внутри rawText
(замена на `<\/review>`), чтобы текст отзыва не мог закрыть разделитель и выдать свои
строки за инструкции. Контракт зафиксирован в prompts/README.md.

**Последствия.** Резка секций — тривиальный код без маппингов enum→заголовок; переименование
заголовков в platforms.md становится breaking change и требует правки кода (отмечено в
README). Прорыв разделителя закрыт на уровне сборки, а не только инструкций в промпте.

## ADR-016: Refresh-токен — opaque random, в БД только sha256-хэш (2026-06-11)

**Контекст.** ТЗ фиксирует refresh 30d в httpOnly cookie, ротацию и детект повторного
использования, но не формат токена. Refresh-JWT не даёт ничего сверх opaque-строки
(состояние всё равно в БД ради ревокации), зато усложняет инвалидацию.

**Решение.** Refresh-токен — `randomBytes(48).hex`; в `RefreshToken.tokenHash` хранится
sha256-хэш (утечка БД не раскрывает действующие токены). Ротация: при refresh старая
запись помечается `revokedAt`, создаётся новая. Повторное использование записи с
`revokedAt != null` → ревокация всех активных токенов пользователя + 401. Кука
`rd_refresh`: httpOnly, sameSite=lax, secure в production, `path=/api/v1/auth`.

**Последствия.** Простая инвалидация (`updateMany`), ноль криптографии сверх sha256;
access-токен остаётся JWT `{ sub, companyId }`. Кука не уходит на не-auth эндпоинты.

## ADR-017: bcryptjs вместо нативного bcrypt (2026-06-11)

**Контекст.** ТЗ требует «bcrypt cost 12». Нативный пакет `bcrypt` тянет node-gyp /
prebuilds и периодически ломает установку в Docker/CI на разных архитектурах.

**Решение.** Используем `bcryptjs` (чистый JS, тот же алгоритм и формат хэша `$2a/b$`),
cost 12 — требование ТЗ выполняется. Фиктивный compare при отсутствии пользователя
делается с заранее посчитанным bcrypt-хэшем случайной строки (timing-safe login).

**Последствия.** `pnpm install` детерминирован на любой платформе. Хэширование медленнее
нативного, но на cost 12 это десятки миллисекунд на запрос auth — приемлемо
(auth и так под rate limit 10 req/min). Замена на нативный bcrypt — drop-in.

## ADR-018: Валидационные правила, не зафиксированные в ТЗ (2026-06-11)

**Контекст.** Контракты задают поля DTO, но не все границы: длина пароля, имени
компании, формат email-нормализации.

**Решение.** В `packages/contracts`: пароль 8–72 символа (72 — предел bcrypt), email
trim + lowercase (уникальность без дублей по регистру), имя компании 1–200 символов,
`toneOfVoice.avoid` ≤ 1000 символов (ADR-011), `pageSize` ≤ 100. PATCH /company/me
требует хотя бы одно поле. Niche и plan через PATCH не меняются (тарифы — вне scope).

**Последствия.** Единые правила на фронте и бэке из одной zod-схемы; демо-пароль
`Demo12345!` проходит. Ужесточение политики пароля — правка одной схемы.

## ADR-019: FakeLlmProvider за env LLM_PROVIDER=fake с маркерами режимов ошибок (2026-06-11)

**Контекст.** Dev/QA-циклу нужен детерминированный конвейер без сети и без расхода
токенов; QA-сценарии требуют воспроизводимых ошибок LLM (невалидный вывод, таймаут,
сетевая ошибка), которые на реальном API не вызвать по требованию.

**Решение.** `FakeLlmProvider` включается env `LLM_PROVIDER=fake` (тот же DI-токен
`LLM_PROVIDER`, код конвейера не меняется). Возвращает детерминированный русский
payload; id похожих отзывов берёт из блока кандидатов system-промпта и добавляет
заведомо чужой id — демонстрирует серверную фильтрацию similarReviewIds. Режимы
ошибок включаются маркерами в тексте отзыва: `[[FAKE:INVALID]]` → LlmInvalidOutputError,
`[[FAKE:TIMEOUT]]` → LlmTimeoutError, `[[FAKE:NETWORK]]` → LlmNetworkError —
управление сценарием на уровне одного запроса, без перезапуска приложения.
При `LLM_PROVIDER=anthropic` env-валидация требует ANTHROPIC_API_KEY на старте.

**Последствия.** Смоук всего конвейера (POST → SSE → DONE/FAILED → компенсация
лимита) гоняется локально и в CI без ключа Anthropic. Маркеры — служебный контракт
fake-режима; в прод-режиме они игнорируются (обычный текст отзыва).

## ADR-020: Топология воркера — embedded в dev, отдельный entrypoint в проде (2026-06-11)

**Контекст.** ТЗ требует BullMQ-воркер и в процессе API (dev-удобство), и отдельным
процессом для прод-образа, но не задаёт механизм переключения.

**Решение.** `GenerationWorkerService.start()` идемпотентен. В процессе API воркер
стартует на bootstrap при `WORKER_EMBEDDED=true` (дефолт — dev работает из коробки).
Прод: API запускается с `WORKER_EMBEDDED=false`, воркер — отдельный entrypoint
`src/worker.ts` (NestFactory.createApplicationContext(AppModule) без HTTP + явный
start()). Один AppModule на оба процесса — отдельного WorkerModule нет.

**Последствия.** Один Docker-образ обслуживает обе роли (разный CMD); масштабирование
воркеров — репликами процесса worker.js. Минус: контекст воркера поднимает и
HTTP-неиспользуемые провайдеры (контроллеры без сервера) — цена принята за простоту.

## ADR-021: Подпись тона `premium` в промпте — «уверенный»; расхождение с COPY.md (2026-06-11)

**Контекст.** Контракт `ToneOfVoiceSchema.tone` (по схеме 02-DEVELOPER) — `soft |
neutral | premium`, а COPY.md и прототипы используют значение `confident`
(«Уверенный»). Блок тона system-промпта требует подписи из COPY.md.

**Решение.** Контракт не меняем (источник истины — 02-DEVELOPER, данные 2.1 уже
живут с `premium`). Сборщик промпта маппит подписи: soft → «мягкий», neutral →
«нейтральный», premium → «уверенный». Расхождение значения `confident` из COPY.md
фиксируется здесь: фронтенд (2.4) обязан отправлять в API значение `premium` для
опции «Уверенный».

**Последствия.** Промпт согласован с терминологией COPY.md без миграции данных.
Если при реализации фронта `premium` окажется неудобным, переименование в
`confident` — отдельный ADR с миграцией JSON-колонки.

## ADR-022: Usage текущего периода в ответе GET /company/me (2026-06-11)

**Контекст.** ТЗ требует «остаток лимита виден всегда» (счётчик в сайдбаре, форма
генерации, настройки, 402-экран), но отдельного эндпоинта usage в контрактах нет.
Фронту нужны used/limit/period при каждой загрузке каркаса приложения.

**Решение.** GET /company/me расширен полем `usage: { used, limit, period }`
(схема `CompanyMeResponseSchema` в contracts; period — «YYYY-MM»). CompanyService
переиспользует UsageService механики лимитов (ADR-002) — цифры берутся из того же
источника, что и при резервировании. Отдельный эндпоинт не вводится: компания и
usage нужны вместе в одних и тех же местах UI, один запрос вместо двух.

**Последствия.** Один TanStack-query `['company','me']` питает сайдбар, форму,
настройки и 402-экран; после DONE/FAILED фронт инвалидирует этот ключ — счётчик
обновляется без поллинга. PATCH /company/me по-прежнему возвращает чистый CompanyDto.

## ADR-023: Удержание узла «Пакет» после события DONE перед показом карточек (2026-06-11)

**Контекст.** Анимация пайплайна событийная (MOTION.md): узлы активируются
SSE-статусами, а финальный статус DONE приходит сразу с payload. Если заменить
панель карточками в момент DONE, четвёртый узел «Пакет» не успевает визуально
прожить свой этап — рейка обрывается на третьем узле.

**Решение.** Событие DONE сначала активирует четвёртый узел (рейка дозаполняется
до 100%), и только через 900 мс панель заменяется карточками результата
(стаггер 80/190/300/410 мс по MOTION.md). При `prefers-reduced-motion: reduce`
удержания нет — карточки показываются сразу и без стаггера.

**Последствия.** Единственный «таймерный» элемент анимации — фиксированная пауза
поверх событийной модели; статусы по-прежнему приходят только из SSE. Цена —
+0,9 с до результата на happy path; принято ради читаемости signature-анимации.

## ADR-024: Данные клиента — ручное опциональное поле authorName в MVP (2026-06-11)

**Контекст.** Фидбэк владельца продукта по экрану результата: не хватает исходного
сообщения клиента, оценки и даты; «хорошо бы видеть данные клиента». Полные данные
клиента (имя, аватар, история отзывов) живут в профилях авторов на площадках
(Яндекс.Карты / 2ГИС / Ozon / Wildberries), но автосбор отзывов с площадок
в MVP не входит (OVERVIEW) — интеграций, из которых профиль можно подтянуть, нет.

**Решение.** В MVP данные клиента — это одно ручное опциональное поле `authorName`
(`Review.authorName String?`, в контракте POST /reviews — optional, trim,
≤ 100 символов). Поле заполняется владельцем при вставке отзыва, показывается
в блоке «Исходный отзыв» и передаётся в user-блок промпта рядом с оценкой
и площадкой (base.md уже велит обращаться по имени, если оно есть). В логи имя
клиента не пишется — как и тексты отзывов. Автоподтягивание профиля автора
из интеграций с площадками — post-MVP и выходит за scope.

**Последствия.** Схема и контракты уже содержат `authorName`: будущая интеграция
с площадками ляжет в это же поле без миграции — изменится только источник
значения (API площадки вместо ручного ввода). Расширение данных клиента
(аватар, ссылка на профиль) потребует отдельных полей и отдельного решения.

## ADR-025: Настраиваемые лимиты throttler через env для E2E-окружения (2026-06-11)

**Контекст.** Playwright E2E global-setup создаёт 5 тестовых пользователей через UI-формы
(регистрация + онбординг) последовательно. Каждый UI-логин/регистрация делает 2–3 запроса
к `/auth/*`. При дефолтном лимите 10 req/60s (настроенном на AuthController) setup
исчерпывал квоту на 3–4 пользователе и получал 429, из-за чего `users.json` оставался
пустым и все 16 тестов падали с `Fixture 'xss' not found. Available: <пусто>`.

**Решение.** Сделать оба throttler-лимита настраиваемыми через переменные окружения:
- `THROTTLE_DEFAULT_LIMIT` / `THROTTLE_DEFAULT_TTL_MS` — глобальный лимит (дефолт 60/60000)
- `THROTTLE_AUTH_LIMIT` / `THROTTLE_AUTH_TTL_MS` — лимит `/auth/*` (дефолт 10/60000)

`ThrottlerModule.forRootAsync` читает значения через `ConfigService`. `AuthController`
использует `process.env.THROTTLE_AUTH_LIMIT` в декораторе `@Throttle` (значение
вычисляется при загрузке модуля — до создания DI-контейнера, поэтому `ConfigService`
недоступен).

E2E-окружение запускает API с `THROTTLE_AUTH_LIMIT=200 THROTTLE_DEFAULT_LIMIT=200`.
Продакшен-дефолты не изменены.

**Последствия.** Продакшен не затронут: без явного env сохраняются дефолты 60/10.
Паттерн «env-override для test» вместо «убрать throttler» — безопаснее и соответствует
принципу защитного кода. При будущем переходе на Redis-backed throttler (TTL-sliding)
те же переменные останутся актуальны.

## ADR-026: Сохранение email пользователя в localStorage при логине (2026-06-11)

**Контекст.** `AuthContext` хранит `email` пользователя в памяти React-состояния —
только если `setSession(token, email)` вызван явно (при логине через UI-форму).
При восстановлении сессии из `rd_refresh`-куки (POST /auth/refresh при загрузке страницы)
email не восстанавливается: JWT содержит только `sub` + `companyId`, без email.
Экран апгрейда (`/app/upgrade`) показывает тост «Записали — напишем на вашу почту» вместо
реального адреса, что ухудшает UX и ломает E2E-тест `CTA-заглушка показывает тост с email`.

**Решение.** При `setSession(token, knownEmail)` сохранять email в `localStorage['rd_email']`.
При bootstrap `AuthProvider` читает `localStorage['rd_email']` и восстанавливает `email`
в React-состояние. При logout — удалять из localStorage. Обращения к `localStorage`
обёрнуты в try/catch (SSR/приватный режим браузера).

**Последствия.** Email пользователя теперь доступен после перезагрузки страницы.
Для MVP это единственное поле сессии, которое не декодируется из JWT и требует отдельного
персистирования. В будущем `/auth/me`-эндпоинт или email в JWT-payload устранят необходимость
в этом слое.

## ADR-027: Хост-порты compose-стенда параметризованы env, дефолты 3200/4200 (2026-06-12)

**Контекст.** 05-DEVOPS целится в «`docker compose up` → приложение на localhost:3000»,
но bare-metal dev-стенд (`pnpm dev`) уже занимает 3000 (web) и 4000 (api); compose-стенд
должен сосуществовать с ним на одной машине (dev, CI, демо).

**Решение.** Все публикуемые порты в `docker/compose.yaml` — через env с дефолтами:
`WEB_PORT=3200`, `API_PORT=4200` (dev-оверрайд дополнительно: `PG_PORT=54325`,
`REDIS_PORT=63793`). Наружу публикуются только web и api; postgres/redis — во внутренней
сети, их порты открывает только `compose.dev.yaml`. Дефолт `NEXT_PUBLIC_API_URL`
в `.env.example` указывает на compose-стенд (`http://localhost:4200/api/v1`);
bare-dev фронтенд использует встроенный фолбэк `:4000` (Next.js не читает корневой .env).
`CORS_ORIGINS` в `.env.example` включает оба origin'а (:3000 и :3200).

**Последствия.** `cp .env.example .env && docker compose -f docker/compose.yaml
--env-file .env up --build` работает на машине с запущенным bare-dev. Желающие
канонический :3000 ставят `WEB_PORT=3000` в .env. Отклонение от буквы ТЗ
(«localhost:3000») зафиксировано здесь.

## ADR-028: Мигратор — отдельный target в api.Dockerfile и 4-й образ в GHCR (2026-06-12)

**Контекст.** `prisma migrate deploy` и seed (`tsx prisma/seed.ts`) требуют dev-зависимостей
(prisma CLI, tsx), которых нет в прод-образе api (<300MB). ТЗ требует одноразовый
migrate-сервис в compose и порядок первого запуска migrate → api/worker → web в Dokploy.

**Решение.** В `docker/api.Dockerfile` добавлен target `migrate` (наследует build-стейдж
с полными node_modules; CMD: `prisma migrate deploy` + seed при `SEED_ON_START=true`).
Compose собирает его через `target: migrate`; CI публикует как 4-й образ
`ghcr.io/<repo>/migrate` (тяжёлый, ~1.5GB — приемлемо: запускается одноразово,
слои кэшируются). Seed включается только явным флагом `SEED_ON_START=true`.

**Последствия.** Один Dockerfile — единый источник для api/migrate (нет дрейфа версий
prisma между миграцией и рантаймом). В Dokploy migrate запускается как сервис и
останавливается после кода выхода 0 (ручной шаг, см. DEPLOY.md).

## ADR-029: CSP веб-фронтенда — в next.config.ts, а не на Traefik (AUDIT-001) (2026-06-12)

**Контекст.** AUDIT-001 (medium): HTML-поверхность web не покрыта CSP. Варианты:
заголовки в `next.config.ts` (`headers()`) или labels на Traefik-уровне.

**Решение.** CSP задаётся в `next.config.ts`: политика версионируется вместе с кодом,
действует во всех окружениях (dev, compose, e2e, Dokploy), а не только за прокси,
и видит `NEXT_PUBLIC_API_URL` для `connect-src`. Политика:
`default-src 'self'; script-src 'self' 'unsafe-inline' (+'unsafe-eval' только в dev);
style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src fonts.gstatic.com;
connect-src 'self' <origin API>; frame-ancestors 'none'; object-src 'none'; base-uri 'self'`
+ X-Content-Type-Options/X-Frame-Options/Referrer-Policy.

**Последствия.** AUDIT-001 закрыт. `script-src 'unsafe-inline'` оставлен осознанно:
App Router использует inline-скрипты, строгий nonce-CSP требует middleware — вне scope
MVP (отдельная задача при ужесточении). Traefik security-заголовки не дублируем —
один источник истины.

## ADR-030: E2E-стенд CI — отдельный docker/compose.e2e.yaml с контрактом QA (2026-06-12)

**Контекст.** CI-джоба e2e должна «поднять compose и прогнать Playwright», но
`apps/web/e2e/global-setup.ts` зашивает контракт QA-стенда: API :4100, Web :3100,
сброс БД через `docker exec rd-e2e-pg psql -U replydesk_e2e -d replydesk_e2e`.

**Решение.** Отдельный `docker/compose.e2e.yaml` (project `replydesk-e2e`):
container_name `rd-e2e-pg`/`rd-e2e-redis`, учётки `replydesk_e2e`, api на :4100
с `LLM_PROVIDER=fake`, embedded-воркером и поднятыми throttler-лимитами (ADR-025),
web на :3100 собирается с build-arg `NEXT_PUBLIC_API_URL=http://localhost:4100/api/v1`.
Несекретные тестовые значения — в коммитнутом `docker/e2e.env`.

**Последствия.** E2E в CI гоняется против прод-образов (тех же Dockerfile, что едут
в GHCR) — сборка проверяется e2e-прогоном. Локально файл не запускать при живом
QA-стенде (конфликт имён контейнеров). Тестовый env-файл в гите — осознанное
исключение из правила «секреты только в .env» (секретов не содержит).

## Зафиксированные мелочи

- Повторное использование старого refresh-токена → ревокация **всех** refresh-токенов
  пользователя (отдельное поле «семьи» в схеме не вводим).
- Код 402 для LIMIT_EXCEEDED оставлен как есть (зафиксирован в ТЗ и тестах).
- Риск 152-ФЗ (тексты отзывов с ПД уходят в зарубежный LLM) принят для MVP; смягчение —
  провайдер-агностичный адаптер `LlmProvider`, позволяющий заменить провайдера без

- Период компенсации лимита передаётся в job BullMQ (зафиксирован в момент
  резервирования): FAILED на границе месяца компенсирует тот период, в котором
  резервировали, а не текущий.
- `similarReviewIds` после фильтрации по кандидатам: пустой список ⇒ `isRepeat`

- Таймаут вызова LLM считается ретраябельной сетевой ошибкой (2 ретрая), после

## ADR-031: Dev-провайдер claude-cli — генерация через локальный Claude Code CLI (2026-06-11)

**Контекст.** Для локальной проверки без моков нужен реальный LLM, но API-ключа
Anthropic в окружении нет; у Дмитрия есть подписка Claude Code (локальная сессия
`claude` аутентифицирована).

**Решение.** Третий вариант `LLM_PROVIDER=claude-cli`: `ClaudeCliProvider` вызывает
`claude -p --output-format json --max-turns 1 --disallowed-tools '*'` (модель из env
CLAUDE_CLI_MODEL, default sonnet). Tool use в headless CLI недоступен — контракт схемы
обеспечивается инструкцией «один JSON по схеме» + zod-валидацией с одним повторным
запросом (та же семантика, что у AnthropicProvider). Таймаут 120с. Ошибки маппятся
в иерархию LlmError.

**Последствия.** Полный happy path с настоящей генерацией локально без ключа.
Строго dev-only: в прод-образах CLI отсутствует; провайдер медленнее SDK
(спавн процесса на вызов) и не для нагрузки. Прод-путь не изменился (anthropic).

## ADR-032: Ниша редактируема через PATCH /company/me + новая ниша HOOKAH (2026-06-12)

**Контекст.** Фидбэк владельца: «не работает выпадающий список ниш в настройках,
если нет „кальяны и бар" — добавь». По исходному ТЗ ниша задавалась один раз
в онбординге и через PATCH /company/me не менялась — селект в настройках был
задизейблен с единственной опцией, что владелец прочитал как баг.

**Решение.** (1) `UpdateCompanyDtoSchema` принимает опциональное `niche`,
CompanyService.update сохраняет его (по-прежнему строго по companyId из JWT);
селект в настройках разблокирован и показывает все ниши. (2) В enum Niche
(Prisma + contracts) добавлено значение HOOKAH («Кальянная и бар»), миграция
`20260611200503_add_niche_hookah`, нишевый промпт `prompts/niches/hookah.md`
(специфика: кальян/угли/вентиляция/бронь; деликатное — 18+ и антитабачное
законодательство РФ: не рекламировать табак в публичных ответах). Онбординг
подхватывает новую нишу автоматически (`NicheSchema.options`).

**Последствия.** Смена ниши применяется к следующим генерациям без рестарта:
пайплайн читает Company из БД на каждый job, кеш промпт-файлов ключуется путём
файла, поэтому `hookah.md` подхватывается при первом обращении. Старые
сгенерированные ответы не пересчитываются.

## ADR-033: Юридические документы и двойное согласие на регистрации (2026-06-12)

**Контекст.** Юрист подготовил четыре документа (152-ФЗ): пользовательское соглашение,
политику обработки ПД, согласие на обработку ПД и отдельное согласие на трансграничную
передачу данных поставщику LLM (Anthropic PBC, США — ст. 12 152-ФЗ требует отдельной
отметки). Нужны публичный просмотр документов и доказуемая фиксация фактов согласий.

**Решение.** (1) Публичный роут `/legal/[slug]` (без авторизации) для четырёх slug'ов —
markdown из `apps/web/content/legal/*.md` читается на сервере (RSC, generateStaticParams,
`dynamicParams=false` → незнакомый slug 404), рендер react-markdown БЕЗ raw-HTML
(никакого dangerouslySetInnerHTML); frontmatter (title/version/updated) парсится без
зависимостей. Ссылки на документы — в блоке «Документы» на /app/settings.
(2) На форме регистрации два РАЗДЕЛЬНЫХ обязательных чекбокса (склеивать нельзя):
«соглашение + политика + согласие на обработку ПД» и отдельный «передача данных в LLM
Anthropic (США)» с кратким предупреждением о трансграничной передаче; кнопка
задизейблена, пока не отмечены оба. (3) Контракт `RegisterDto` расширен
`acceptTerms: literal(true)` и `acceptLlm: literal(true)` — отсутствие/false → 422.
(4) Доказуемость: поля `User.consentPdAt`, `consentLlmAt`, `consentDocsVersion`
(миграция `20260611203353_add_user_consent_fields`); AuthService.register пишет
момент согласия и версию редакции (константа `CONSENT_DOCS_VERSION='v1.0'`,
синхронна с frontmatter документов). В логи факты согласий не пишутся — только в БД.

**Последствия.** Регистрация без двух согласий невозможна ни через UI, ни через API.
Старые пользователи имеют null в consent-полях (повторного подтверждения для MVP не
требуем). При новой редакции документов поднять `version` во frontmatter и
`CONSENT_DOCS_VERSION`; механизм пере-акцепта существующих пользователей — вне scope MVP.

## ADR-034: LLM_PROVIDER=auto — автоматический выбор провайдера (2026-06-12)

**Контекст.** Владелец просил не настраивать провайдера вручную: при наличии API-ключа
использовать Anthropic API, без ключа локально — claude-cli (ADR-031). Явный
`LLM_PROVIDER=anthropic` как дефолт заставлял каждый dev-стенд править env.

**Решение.** Новое значение `LLM_PROVIDER=auto` становится дефолтом. Резолв в фабрике
LlmModule: непустой `ANTHROPIC_API_KEY` → AnthropicProvider, иначе при наличии бинаря
`claude` в PATH → ClaudeCliProvider; нет ни ключа, ни CLI → падение на старте с
сообщением «нет ANTHROPIC_API_KEY и claude CLI недоступен» (подсказка: задать ключ,
поставить CLI или LLM_PROVIDER=fake). Выбор логируется на старте (без значения ключа).
Пустая строка `ANTHROPIC_API_KEY=` трактуется как отсутствие ключа (transform в env.ts).
Явные anthropic/fake/claude-cli работают как раньше (anthropic без ключа — ошибка на
старте). Если бинарь исчез после старта, первая генерация падает в FAILED с читаемой
ошибкой «claude CLI не найден в PATH» (лимит компенсируется штатно).

**Последствия.** Локальный dev работает «из коробки» по подписке Claude Code; прод
задаёт ключ — auto безопасно выбирает anthropic. E2E/CI не затронуты: compose.e2e и
CI задают `LLM_PROVIDER=fake` явно. В прод-контейнере без ключа API не стартует
(понятная ошибка вместо тихой деградации).

## ADR-035: Тарифная сетка биллинга и источник цен (2026-06-12)

**Контекст.** Биллинг (ЮKassa) вводит платные подписки START (100 ген/мес) и
BUSINESS (1000 ген/мес) на 1/3/6/12 месяцев. Цены нигде не были зафиксированы
(ADR-012 осознанно их избегал). Ориентир владельца: себестоимость генерации
×3 ≈ 10 ₽/генерация на START.

**Решение.** Цены задаются env-переменными `PRICE_{PLAN}_{N}M` / `PRICE_PACK_{N}`
(копейки) с дефолтами: START — 790/2190/3990/6990 ₽ за 1/3/6/12 мес;
BUSINESS — 5990/16490/29990/53990 ₽. Длинные периоды дают скидку ~8–27%
относительно помесячной цены. FREE (10 ген/мес) остаётся бесплатным и подпиской
не оформляется. Источник цен — env (не БД): смена прайса — перезапуск без
миграции; цена зафиксированного периода хранится в `Subscription.price` и
`PaymentTransaction.amount`, поэтому смена env не влияет на уже оплаченные
периоды и расчёт возвратов.

**Последствия.** Один источник истины для цен на бэке; фронт получает цены
через будущий публичный конфиг или хардкод копидека (вне scope этого шага).
Изменение сетки = правка env + .env.example.

## ADR-036: Пакеты генераций вместо микросписаний + pro-rata возврат по фактическим дням (2026-06-12)

**Контекст.** Поштучная оплата генераций (10 ₽) экономически бессмысленна:
минимальная комиссия эквайринга съедает маржу, а ЮKassa не предназначена для
микроплатежей. Также нужна отмена подписки с честным возвратом.

**Решение.** (1) Разовые пакеты 10/50/100 генераций (100/500/1000 ₽) —
один платёж, начисление в `Company.packageCredits`; пакетные генерации НЕ
сгорают и не имеют автопродления. (2) Отмена подписки (POST /billing/cancel) —
pro-rata возврат: `остаток_дней / всего_дней × сумма_последнего_платежа`,
где «всего дней» считается по фактическому отрезку (expiresAt − paidAt), а не
по 30×месяцев — иначе на 31-дневных месяцах возврат превышал оплату и ЮKassa
отвечала 400 (грабли перенесены из habby-tracker). Сумма дополнительно
cap'ится исходным платежом. Отмена — compare-and-swap ACTIVE→EXPIRED
(двойной клик не даёт второго возврата); при отказе ЮKassa в возврате
подписка и план откатываются (пользователь не остаётся без доступа и без денег).

**Последствия.** Каждый платёж ≥ 100 ₽ — комиссия терпима; неиспользованные
пакетные кредиты — обязательство перед клиентом (при отмене подписки пакеты
не возвращаются, они отделены от подписки). Возврат идёт на последний
успешный платёж подписки (для продлений — корректно: новый период оплачен им).

## ADR-037: Порядок списания генераций — лимит тарифа → пакет → 402; компенсация в источник (2026-06-12)

**Контекст.** С появлением пакетов у компании два источника генераций:
месячный лимит тарифа (UsageCounter, ADR-002) и пакетные кредиты
(Company.packageCredits). Нужен детерминированный порядок и совместимость
с моделью «резервирование при POST + компенсация при FAILED».

**Решение.** `UsageService.reserve()` сначала пытается атомарно занять
месячный лимит (условный UPDATE `used < limit`), при неудаче — атомарный
декремент пакета (`updateMany ... packageCredits >= 1`); если пусто и там —
402 LIMIT_EXCEEDED. reserve() возвращает источник (`PLAN | PACKAGE`),
который фиксируется в job BullMQ рядом с периодом; компенсация при FAILED
возвращает генерацию ровно туда, откуда списали (PLAN — декремент счётчика,
PACKAGE — инкремент кредитов). Старые job без поля источника трактуются
как PLAN.

**Последствия.** Месячный лимит «сгорает» первым (он обнуляется новым
периодом), пакет тратится только сверх лимита — выгодно пользователю.
Конкурентность не пробивает ни лимит, ни пакет (оба списания — условные
UPDATE на блокировке строки). Тест «11-я генерация FREE с пакетом проходит
из пакета» закреплён интеграционно.

## ADR-038: Webhook ЮKassa без подписи — перепроверка через GET + CAS-идемпотентность (2026-06-12)

**Контекст.** HTTP-уведомления ЮKassa не подписываются (аутентификация — только
IP-allowlist на их стороне), endpoint обязан быть публичным (без JWT). Любой
может отправить поддельный `payment.succeeded` на наш URL.

**Решение.** POST /billing/webhook — @Public; defense-in-depth по образцу
habby-tracker: (а) доверяем не телу уведомления, а перепроверке — GET
/v3/payments/{id} (для refund.succeeded — GET /v3/refunds/{id}) с нашими
Basic Auth-ключами; несовпадение статуса → событие отброшено (200, чтобы
ЮKassa не ретраила вечно), ошибка GET → 500 (ЮKassa повторит доставку);
(б) идемпотентность применением — compare-and-swap `updateMany
{status: PENDING} → SUCCEEDED`: повторная доставка видит count=0 и выходит,
подписка не продлевается дважды. Активация подписки: продление от текущего
expiresAt, если подписка того же плана активна (предоплата не съедает дни),
иначе от now; `payment_method` может не сохраниться — graceful null (маска
старой карты затирается, чтобы UI не врал). Отвязка карты — только локально:
у API ЮKassa нет delete-endpoint для payment_method. Автопродление —
POST /billing/cron/renewals (Bearer CRON_SECRET, timing-safe сравнение):
горизонт 1 час, race-защита 15 минут через недавнюю PENDING-транзакцию,
Idempotence-Key `renewal:{subId}:{expiresAt}` — повторный cron бьёт в тот же
платёж ЮKassa. Без YOOKASSA_SHOP_ID/SECRET_KEY биллинг выключен: checkout
отдаёт 503 BILLING_DISABLED, остальные эндпоинты /billing работают.

**Последствия.** Поддельные webhook бессильны (проверено интеграционным
тестом); цена — один GET к ЮKassa на уведомление. ЮKassa-клиент — свой
fetch-обёртка (~80 строк, инжектируемый класс) вместо npm-пакета: в тестах
подменяется через DI, наружных зависимостей нет.

## ADR-039: Прайс на фронте — статичная таблица констант, синхронизируемая с env API (2026-06-12)

**Контекст.** Странице /app/billing нужны цены подписок и пакетов до покупки,
но `GET /billing` (BillingOverview) прайс не отдаёт: цены живут в env API
`PRICE_{PLAN}_{N}M` / `PRICE_PACK_{N}` (ADR-035). Варианты: добавить публичный
эндпоинт прайса или зашить таблицу на фронте.

**Решение.** Статичная таблица в одном файле
`apps/web/src/lib/billing/prices.ts` (копейки, дефолты env из ADR-035)
с комментарием «должно совпадать с env API». Новый эндпоинт не добавляем:
прайс меняется редко, оба изменения (env и константа) и так требуют деплоя,
а лишний публичный эндпоинт — лишняя поверхность. Цена в checkout всё равно
считается бэкендом из env — расхождение таблицы с env искажает только
витрину, не списание. Производные «цена в месяц» и «выгода N%» считаются
на фронте от помесячной цены.

**Последствия.** При смене цен в env обязательна синхронная правка
`prices.ts` (и наоборот). Если прайс станет динамическим (акции,
персональные скидки) — переезжаем на отдаваемый API прайс одним полем
в BillingOverview.

## ADR-040: Self-heal сверка PENDING-платежей при открытии страницы тарифа (2026-06-12)

**Контекст.** Активация оплаты приходит вебхуком ЮKassa, но вебхук не доставляется
на локальный стенд (нет публичного URL) и может теряться на проде (сбой доставки,
даунтайм). Пользователь оплатил — а тариф «висит» в обработке.

**Решение.** Паттерн habby-tracker (reconcilePendingPayments): при GET /billing
PENDING-транзакции компании за последние 24 часа (до 5 штук) сверяются с ЮKassa
через GET /v3/payments/{id} и применяются тем же кодом, что и вебхук
(applySucceededPayment — CAS-идемпотентность сохраняется). Ошибки сверки логируются
и не валят страницу.

**Последствия.** Локальное тестирование оплаты работает без публичного вебхука;
на проде — страховка от потерянных вебхуков. Цена — до 5 запросов к ЮKassa при
открытии страницы тарифа с незакрытыми платежами.
